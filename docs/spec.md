# Спецификация: Ветер / ЕХАТЬ? (Android-приложение для головного устройства Denza N9)

> Документ — единственный источник истины. Читать целиком перед началом работы.
> Это сохранённая копия исходной спецификации, переданной для реализации.

---

## 0. Контекст и цель
Единственный пользователь — мужчина ~55 лет, живёт в Актау (Мангистауская область, Казахстан). Регион с постоянными сильными ветрами. Есть загородный дом у моря примерно в часе езды.

Сейчас он открывает rp5.kz и **сам интерпретирует цифры**, чтобы понять: ехать сегодня на дачу или нет, можно ли на море.

**Цель приложения: убрать шаг интерпретации.** Приложение отвечает на вопрос, а не показывает данные. Цифры остаются — но вторым планом.

Это не клон rp5. Это не «погодное приложение». Это кнопка «ехать?».

### Приёмочный критерий (единственный, который важен)
Пользователь садится в машину, бросает взгляд на экран **менее чем на 2 секунды** и получает ответ. Если ему нужно читать, вчитываться или скроллить — приложение провалено.

## 1. Anti-goals (чего НЕ делать)
Явный запрет: авторизация/аккаунты/онбординг/splash-экран-заставка, аналитика/крашлитика/телеметрия, Firebase, Google Play Services, push, карты, выбор городов/поиск локаций, реклама/монетизация, светлая тема, лишние анимации, туториалы.

## 2. Технические ограничения
Целевое устройство — головное устройство BYD/DiLink на кастомном Android. Это **не** Android Automotive OS.

- minSdk 24. Проверить `adb shell getprop ro.build.version.release`.
- Google Play Services отсутствуют. Разработка на AOSP-образе.
- Ориентация только landscape.
- Runtime permissions — ноль. Координаты захардкожены. INTERNET — normal.
- Сеть ненадёжна. Оффлайн-first обязателен.
- Взаимодействие боковым зрением. Тач-таргеты ≥ 64dp. Основные цифры ≥ 72sp.

### Стек
Kotlin, Jetpack Compose (Material3), OkHttp + kotlinx.serialization, DataStore Preferences, kotlinx.coroutines, JUnit + kotlin.test. Без DI-фреймворков — ручная сборка графа. Версии — через `libs.versions.toml`, актуальный Compose BOM.

---

## 3. Архитектура
Один Gradle-модуль `:app`, три пакета: `domain/` (чистый Kotlin, ноль android-импортов, здесь все тесты), `data/`, `ui/`.

Правило: `domain/` не импортирует `android.*`/`androidx.*` — тесты запускаются как обычные JVM-тесты без Robolectric.

---

## 4. Источник данных — Open-Meteo
GET `https://api.open-meteo.com/v1/forecast` с hourly (wind_speed_10m, wind_gusts_10m, wind_direction_10m, temperature_2m и др.), current, `wind_speed_unit=ms`, `timezone=Asia/Aqtau`, `forecast_days=3`. Обе точки одним запросом (координаты через запятую → массив).

Координаты: AKTAU (43.6481, 51.1722); DACHA — заполнить реальными (в одном месте, `Constants`/BuildConfig).

Волны на Каспии: **проверить запросом** Marine API. Если null — не изобретать эвристику; либо не показывать волны, либо честная подпись «оценка по ветру».

Часовой пояс Asia/Aqtau = UTC+5, без перехода на летнее время. Не полагаться на системный TZ устройства.

## 5. Доменная логика
### 5.1 Статус (светофор)
Считается по порывам `wind_gusts_10m`. `CALM/WINDY/HARSH`. Дефолты порогов: windy 8, harsh 14 м/с. Пороги — в настройках. Доступность: цвет + текст + иконка (дальтонизм).

### 5.2 Прогноз на момент прибытия
`forecastOnArrival(hourly, now, travelDuration=60m)` → ближайший часовой слот к `now + travel`. Карточка «Дача» показывает это значение с подписью «к приезду, ~HH:00».

### 5.3 Встречный ветер по маршруту
`wind_direction_10m` — направление, откуда дует (метео). `headwindComponent = v*cos(windFrom-bearing)` (положительное = встречный). `crosswindComponent = v*sin(...)`. Потеря запаса хода — ЭВРИСТИКА: `rangePenaltyPercent`, аэродоля ~0.6. Подпись «ориентировочно». `routeBearingDeg` — константа.

### 5.4 Кнопка «Поехали?»
«Да» / «Не сегодня» + строка обоснования. Для HARSH — искать ближайшее окно ≥3ч подряд CALM.

---

## 6. UI
Один экран, landscape, тёмная тема. Штамп свежести обязателен; >3ч — оранжевый + «нет связи». Гистограмма 72ч, цвет = статус часа. Стрелка — реальный поворот. Настройки: пороги + время в пути. Скрытый debug-экран по долгому тапу (2с) по заголовку: время последнего запроса, HTTP-статус, ошибка, сырой JSON, «Обновить сейчас».

---

## 7. Оффлайн и кэш
`ForecastState`: Fresh/Stale(>3ч)/Empty. Кэш рисуется до сети. Ошибка при наличии кэша → кэш + оранжевый штамп, без диалога. Ошибка без кэша → пустое состояние с «Повторить». Таймауты 10s/10s. Автообновление раз в 15 мин через LaunchedEffect+delay. Без WorkManager.

---

## 12. Дизайн-спека
Приложение «видит то же, что человек за лобовым стеклом». Токены: Calm #2DD4BF, Windy #FBBF24, Harsh #FB6B4B, Bg #0B1015, Surface, Stroke, Ink, InkDim. Палитра teal/amber/coral вместо красно-зелёной (дальтонизм). Типографика: NumberStyle 96sp Bold, `tnum` обязателен, минимум Bold, контраст ≥7:1. Фон-ветер: один Canvas, ~150 частиц, вектор движения = directionFromDeg+180, withFrameNanos, альфа ≤0.2, цвет = статус, интерполяция цвета 900мс. Ambient glow снизу, alpha ~0.18. Движение: цифры animateIntAsState(400), стрелка spring, цвет/glow tween(900), гистограмма волна, CTA scale 0.985. Стрелка — кратчайшая дуга `((to-from+540)%360)-180`. Запрещено: Light/Thin, чистый чёрный, красно-зелёная пара, спиннер, диалоги ошибок, serif, тонкие иконки, тач <64dp, анимации >900мс.

## 13. Обработка ошибок сети
Ни диалога, ни тоста, ни спиннера поверх данных. Тексты называют факт: «Нет связи. Данные от 11:40». Таксономия FetchError: Offline, Timeout, Server(5xx), Client(4xx), RateLimited(429), Malformed, CaptivePortal. Captive portal — проверять Content-Type до парсинга. Расхождение часов — брать время из заголовка Date, инжектировать Clock. Ретраи: Timeout/Server/RateLimited; не Offline/Client/Malformed/CaptivePortal; максимум 3, бэкофф 1/2/4с ±30% джиттер, RateLimited 30с/Retry-After. Single-flight через Mutex+Deferred. Автовосстановление через ConnectivityManager (без GMS), отписка в onStop. Битые данные: List<Double?>, пропуск столбика, ближайший непустой слот к приезду, короткий массив → Malformed. Матрица состояний UI по свежести (3ч/12ч). Splash — системный, keepOnScreen пока не прочитан кэш, без анимаций/задержек.

## 14. Модель факторов (приоритет над §5.1, §5.4, §6)
Binding constraint — ограничивающий фактор определяет вердикт и цвет сцены. Ровно ШЕСТЬ факторов: WIND (wind_gusts_10m), HEAT (apparent_temperature), UV (uv_index), PRECIP (precipitation_probability + weather_code), DUST (pm10, air-quality-api), SEA (wave_height + sea_surface_temperature, marine-api). Три endpoint'а, три параллельных запроса; отказ одного не роняет остальные; DUST/SEA при отказе → relevant=false. Релевантность: UV только днём, SEA только май–сентябрь; нерелевантный фактор не показывается. Пороги в настройках; HEAT — двусторонний; гроза (95–99) → HARSH. Вердикт: худший релевантный фактор, тай-брейк по PRIORITY [PRECIP, DUST, WIND, HEAT, UV, SEA]. Все факторы — на момент прибытия (Дача) / current (Актау). Окно затишья — все релевантные CALM ≥3ч. UI: крупная цифра = ветер, цвет сцены = вердикт (возможна зелёная цифра на красном экране). Полоса факторов: binding увеличен со значением, остальные — точки статусного цвета, фиксированный порядок. Гистограмма — по вердикту часа. Название: «ЕХАТЬ?».
